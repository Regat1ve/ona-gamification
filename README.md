# ona-gamification

Исследовательский инструмент для адресной геймификации на основе организационного сетевого анализа (Organizational Network Analysis, ONA). Программа моделирует коммуникации внутри малой команды, диагностирует структурные аномалии графа взаимодействий, подбирает игровые механики под выявленные проблемы и проверяет статистическую значимость прироста эффективности относительно неадресной базовой линии.

Реализация на Python. Использует только открытые научные библиотеки. Запускается на одном CPU, занимает несколько секунд.

## Что внутри

- **`prototype.py`** — основной модуль. Один прогон в фиксированной канонической конфигурации (25 сотрудников в трёх отделах, четырёхнедельное окно наблюдения). Выполняет полный цикл: генерация логов, построение направленного взвешенного графа, расчёт ONA-метрик (степенная и посредническая центральность, кластеризация), диагностика аномалий, подбор механик из встроенной матрицы соответствия, симуляция адресной интервенции и неадресной базовой линии, повторный замер метрик, статистическая проверка через парный t-критерий и U-критерий Манна — Уитни.
- **`experiments.py`** — параметризованная версия `prototype.py`. Запускает три серии прогонов анализа чувствительности (по размеру команды, по зерну генератора, по плотности коммуникаций) плюс отдельный прогон альтернативной конфигурации с четырьмя отделами. Логика порогов диагностики и формула KPI идентичны `prototype.py`.
- **`prototype_summary.json`** — автоматически формируемая сводка результатов канонического прогона: ONA-метрики до и после, выявленные аномалии, рекомендованные механики, значения статистических тестов.
- **`experiments_summary.json`** — автоматически формируемая сводка всех серий прогонов чувствительности и альтернативной конфигурации.

Все JSON-файлы содержат русскоязычные ключи и сохраняются в UTF-8.

## Зависимости

- Python 3.10 или выше
- `numpy`, `networkx`, `pandas`, `scipy`

Установка:

```bash
pip install numpy networkx pandas scipy
```

## Запуск

Канонический прогон:

```bash
python prototype.py
```

Анализ чувствительности и альтернативная конфигурация:

```bash
python experiments.py
```

Если в консоли неправильно отображается кириллица (актуально для Windows):

```bash
PYTHONIOENCODING=utf-8 python prototype.py
PYTHONIOENCODING=utf-8 python experiments.py
```

## Что регулируется

### `prototype.py`

Все ключевые параметры сосредоточены в верхней части файла как константы модуля.

| Константа | Значение по умолчанию | Описание |
|---|---|---|
| `SEED` | `42` | Зерно генератора псевдослучайных чисел. Полностью определяет результат прогона. |
| `DEPARTMENTS` | `{"Разработка": 10, "Продажи": 8, "Поддержка": 7}` | Состав отделов и их размеры. |
| `WEEKS` | `4` | Длина наблюдаемого окна в рабочих неделях. |
| `DAYS_PER_WEEK` | `5` | Число рабочих дней в неделе. |
| `P_INTRA` | `0.45` | Базовая вероятность контакта между сотрудниками одного отдела за день. |
| `P_INTER` | `0.03` | Базовая вероятность контакта между сотрудниками разных отделов. |
| `ISOLATED_FACTOR` | `0.1` | Множитель активности для сотрудников, помеченных как изолированные. |
| `HUB_FACTOR` | `2.8` | Множитель активности для сотрудников, помеченных как перегруженные посредники. |
| `WEAK_PAIR` | `{"Разработка", "Продажи"}` | Пара отделов с искусственно ослабленной межотдельной коммуникацией. |
| `WEAK_PAIR_FACTOR` | `0.25` | Коэффициент ослабления для `WEAK_PAIR`. |

Чтобы изменить зерно, отредактируйте строку:

```python
SEED = 42
```

Чтобы изменить состав команды, измените словарь `DEPARTMENTS`. Сумма размеров отделов задаёт общее число сотрудников. Параграфы кода, помечающие конкретных сотрудников как изолированных и перегруженных посредников (`employees[2]`, `employees[14]`, `employees[0]`, `employees[10]`), привязаны к индексам в массиве сотрудников и предполагают конкретные размеры отделов; при существенном изменении `DEPARTMENTS` имеет смысл скорректировать и эти индексы или перейти на использование `experiments.py`, где смещения вычисляются автоматически.

### `experiments.py`

Параметры передаются через объект `ExperimentConfig` (декларация в верхней части модуля). Поля:

| Поле | Описание |
|---|---|
| `name` | Идентификатор прогона (используется в выводе и в JSON-сводке). |
| `departments` | Словарь «название отдела → размер». |
| `isolated_idx` | Список индексов сотрудников, помечаемых как изолированные. |
| `hub_idx` | Список индексов сотрудников, помечаемых как перегруженные посредники. |
| `p_intra`, `p_inter` | Базовые вероятности внутригрупповых и межгрупповых контактов. |
| `isolated_factor`, `hub_factor` | Множители активности для специальных ролей. |
| `weak_pair` | Кортеж из двух названий отделов с ослабленной межотдельной связью; `None`, если ослабление не используется. |
| `weak_pair_factor` | Коэффициент ослабления для `weak_pair`. |
| `seed` | Зерно генератора. |
| `weeks`, `days_per_week` | Длина наблюдаемого окна. |

Готовые серии прогонов формируются функциями `run_sensitivity_size`, `run_sensitivity_seed`, `run_sensitivity_p` и `run_alternative_config`. Чтобы добавить свою конфигурацию — создайте `ExperimentConfig` напрямую или через вспомогательный конструктор `make_cfg(name, departments, ...)`, передающий разумные значения по умолчанию для индексов изолятов и хабов.

Пример пользовательского прогона:

```python
from experiments import ExperimentConfig, run_full_experiment

cfg = ExperimentConfig(
    name="custom",
    departments={"Front": 6, "Back": 6, "QA": 4},
    isolated_idx=[2],
    hub_idx=[0],
    p_intra=0.4,
    p_inter=0.04,
    weak_pair=("Front", "QA"),
    seed=123,
    weeks=4,
)
result = run_full_experiment(cfg)
print(result["mean_delta_targeted"], result["mean_delta_universal"])
```

## Что в выходных JSON

`prototype_summary.json` содержит:

- общие параметры прогона (`team_size`, `departments`, `weeks`);
- число сгенерированных событий взаимодействия `n_logs_before`;
- карту выявленных аномалий `findings`;
- межотдельные разрывы `gaps`;
- средний прирост модельного KPI в адресном (`mean_delta_targeted`) и базовом (`mean_delta_universal`) сценариях;
- результаты статистических тестов: парный t-критерий по KPI для обоих сценариев, U-критерий Манна — Уитни (адресный против базового, односторонняя альтернатива), парный t-критерий по суммарной степенной центральности.

`experiments_summary.json` содержит четыре раздела (`sensitivity_size`, `sensitivity_seed`, `sensitivity_p`, `alternative_config`), внутри каждого — массив объектов с теми же полями, что и в индивидуальном прогоне.

## Воспроизводимость

При фиксированных значениях `SEED` (в `prototype.py`) и `seed` (в `ExperimentConfig`) программы детерминированы. Канонический прогон `prototype.py` при `SEED = 42` повторяется бит-в-бит между запусками. Прогоны `experiments.py` детерминированы внутри каждого `ExperimentConfig`; при добавлении новых конфигураций или изменении порядка вызовов конкретные числа могут смещаться, что нормально и не свидетельствует об ошибке.

Изменение `SEED` приводит к новому шаблону связей и к другим конкретным числам, но направление эффекта (адресный сценарий даёт более высокий прирост, чем неадресный) сохраняется при всех проверенных в `experiments.py` значениях.
